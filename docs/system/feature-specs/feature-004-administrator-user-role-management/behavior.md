## User Workflows

### Main workflow: просмотр пользователей

1. Administrator открывает вкладку `Пользователи`.
2. Система должна проверить, что текущий пользователь имеет роль `administrator`.
3. Система должна предоставить список пользователей для управления ролями.
4. Система должна показать для каждого пользователя идентифицирующую подпись, текущую операционную роль и состояние блокировки.
5. Система должна позволить administrator найти пользователя через экранный поиск или фильтр, если список уже получен.

### Main workflow: назначение роли `barista`

1. Administrator выбирает активного пользователя в списке.
2. Система должна предоставить действие назначения роли.
3. Administrator выбирает роль `barista`.
4. Система должна проверить право текущего пользователя на административное управление ролями.
5. Система должна проверить, что назначаемая роль входит в разрешенный набор `barista` / `administrator`.
6. Система должна сохранить роль `barista` для целевого пользователя как операционную роль внутреннего административного контура.
7. Система должна пересчитать доступ целевого пользователя к вкладкам внутреннего административного контура.
8. Система должна вернуть обновленное представление целевого пользователя.

### Blocked workflow: назначение роли `administrator`

1. Administrator выбирает активного пользователя в списке.
2. Administrator выбирает роль `administrator`.
3. Система должна иметь финальное guard-правило, определяющее, кто имеет право назначать роль `administrator`.
4. Текущее состояние источников не позволяет финализировать шаг сохранения роли `administrator`.

### Alternative workflow: список пуст

1. Administrator открывает вкладку `Пользователи`.
2. Система должна вернуть пустой список пользователей, если пользователи для отображения отсутствуют.
3. Система должна показать пустое состояние списка.

### Exception workflow: недостаточная роль текущего пользователя

1. Пользователь без роли `administrator` пытается открыть вкладку `Пользователи` или выполнить назначение роли.
2. Система должна отказать в доступе к административному управлению пользователями.
3. Система должна сохранить роли целевого пользователя без изменений.

### Exception workflow: недопустимая назначаемая роль

1. В систему поступает запрос назначения роли вне набора `barista` / `administrator`.
2. Система должна отклонить изменение роли.
3. Система должна сохранить роли целевого пользователя без изменений.

### Exception workflow: целевой пользователь не найден

1. Administrator отправляет назначение роли для пользователя, который больше не существует в доступном списке.
2. Система должна отклонить изменение роли.
3. Система должна вернуть состояние ошибки для повторного чтения актуального списка.

## Entity View

### Entities

- `User`
- `Role`
- `BootstrapAdministrator`
- `AccessChannel`

### Relations

- `administrator` управляет операционной ролью другого `User`.
- `User.roles` определяют доступ пользователя к вкладкам внутреннего административного контура.
- `BootstrapAdministrator.telegramId` задает главного `administrator` из `ADMIN_TELEGRAM_ID`.

### Invariants

- Система должна хранить роль пользователя как значение из набора `customer`, `barista`, `administrator`.
- Система должна использовать роль `administrator` как условие доступа к вкладке `Пользователи`.
- Система должна пересчитывать доступ к вкладкам после успешного изменения операционной роли.
- Система должна сохранять роль `customer`, если она уже есть у целевого пользователя.
- Система должна поддерживать доступ `barista` только к вкладкам `Заказы` и `Доступность`.
- Система должна поддерживать доступ `administrator` к вкладкам `Заказы`, `Доступность`, `Меню`, `Пользователи`, `Настройки`.

### Identity and ownership

- `User.userId` является внутренним идентификатором пользователя.
- `User.telegramId` является внешним идентификатором пользователя Telegram.
- Серверный контур идентификации и доступа является source of truth для ролей и признака блокировки.
- Клиентский экран `Пользователи` показывает состояние, полученное из серверного source of truth.

## Input Constraints

### Required inputs

- Текущий пользователь administrator.
- Целевой пользователь.
- Назначаемая роль.

### Allowed values

- Назначаемая роль: `barista`, `administrator`.

### Cross-field constraints

- Система должна принимать назначаемую роль только для существующего целевого пользователя.
- Система должна применять изменение роли только после подтверждения административного права текущего пользователя.
- Система должна применять назначение роли `administrator` только после определения финального guard-правила для этой операции.

### Boundary values

- Пустой список пользователей.
- Повторное назначение той же роли.
- Устаревший целевой пользователь после обновления списка.
- Целевой пользователь с ролью `customer`.
- Целевой пользователь с признаком `blocked=true`.

## Validations

### Field validations

- Система должна требовать идентификатор целевого пользователя.
- Система должна требовать назначаемую роль.
- Система должна принимать назначаемую роль только из набора `barista` / `administrator`.

### Business validations

- Система должна проверять существование целевого пользователя.
- Система должна сохранять существующую роль `customer` при назначении операционной роли backoffice.
- Система должна пересчитывать доступ к вкладкам на основании обновленного набора ролей.

### Role or capability validations

- Система должна требовать роль `administrator` для просмотра вкладки `Пользователи`.
- Система должна требовать роль `administrator` для назначения роли `barista`.
- Система должна иметь отдельное финальное правило для назначения роли `administrator`.

## Errors

### User-facing errors

- `administrator-role-required` — Система должна показать отказ в доступе к управлению пользователями.
- `role-not-assignable` — Система должна показать, что выбранная роль не может быть назначена.
- `user-not-found` — Система должна показать, что целевой пользователь недоступен, и предложить обновить список.

### System errors

- Ошибка чтения списка пользователей — Система должна сохранить текущий пользовательский контекст и показать состояние ошибки списка.
- Ошибка сохранения роли — Система должна сохранить локальное представление целевого пользователя в последнем подтвержденном состоянии.

### Error mapping

| Condition                                               | User-visible outcome                                                           | Source                                                           |
| ------------------------------------------------------- | ------------------------------------------------------------------------------ | ---------------------------------------------------------------- |
| Текущий пользователь не имеет `administrator`           | Система должна показать отказ в доступе к вкладке или операции.                | `user-role-and-blocking-management.md`, `identity-and-access.md` |
| Назначаемая роль не равна `barista` или `administrator` | Система должна показать ошибку недопустимой роли.                              | `user-role-and-blocking-management.md`                           |
| Целевой пользователь отсутствует                        | Система должна показать ошибку недоступного пользователя.                      | `user-role-and-blocking-management.md`                           |
| Назначение `administrator` без финального guard-правила | Система должна удерживать сценарий вне architecture handoff до снятия blocker. | `FEATURE-004` blocker, `identity-and-access.md`                  |

## Edge Cases

- Система должна возвращать тот же результат доступа при повторном назначении уже имеющейся операционной роли.
- Система должна пересчитывать вкладки для пользователя, который получит роль `barista`.
- Система должна пересчитывать вкладки для пользователя, который получит роль `administrator`, после снятия blocker по праву назначения.
- Система должна сохранять блокировку пользователя как отдельное состояние, если целевой пользователь уже имеет `blocked=true`.
- Система должна показывать пустое состояние при отсутствии пользователей в списке.

## Scope Constraints

- Блокировка пользователя находится вне scope `FEATURE-004`.
- Разблокировка пользователя находится вне scope `FEATURE-004`.
- Создание нового пользователя находится вне scope `FEATURE-004`.
- Снятие роли `barista` находится вне scope `FEATURE-004`.
- Изменение роли `customer` находится вне scope `FEATURE-004`.
- Архитектурная декомпозиция дочерних задач находится вне scope системного аналитика.
- Production-код находится вне scope этой task-card.

## Safety Constraints

- Роли и capabilities пользователя являются серверным source of truth.
- Клиентская видимость действия назначения роли не является достаточным authorization guard.
- Прямой доступ к вкладке `Пользователи` должен проходить server-aligned capability guard.
- Test-mode bypass допустим только в test environment при серверно разрешенном режиме.
- Назначение роли `administrator` должно иметь разрешенное финальное guard-правило до implementation handoff.
