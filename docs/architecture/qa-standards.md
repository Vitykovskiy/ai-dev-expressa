# Стандарт QA

## Назначение

Этот документ задаёт обязательный стандарт для задач `QA-*`. Он определяет зону ответственности `QA`, требования к e2e-покрытию по `FEATURE-*`, правила передачи результата и порядок взаимодействия с `DO`.

## Область действия

- Стандарт обязателен для всех задач `QA-*`.
- Каждая задача `QA-*` должна быть привязана к одному `FEATURE-*`.
- `QA` не описывает реализацию, не меняет конвейер и не фиксирует карту развёртывания; эти факты живут в контурной карте `application-map/quality-and-delivery.md` и `deployment-map.md`.

## Зона ответственности `QA`

- `QA` проектирует и поддерживает e2e-сценарии для проверяемой фичи.
- `QA` проверяет пользовательские сценарии, негативные ветки, граничные условия и регрессии.
- `QA` фиксирует наблюдаемый результат проверки и передаёт его дальше как готовый артефакт.
- `QA` не владеет выпуском, секретами, маршрутами развёртывания и операциями отката.

## Требования к e2e по `FEATURE-*`

- На каждую `FEATURE-*` должна существовать явная e2e-проверка или набор проверок, покрывающий её критерии приёмки.
- E2e должны быть стабильными, воспроизводимыми и изолированными от ручных действий, если только ручной шаг не является частью утверждённого сценария.
- E2e должны покрывать как успешный сценарий, так и ожидаемые отказы, если это важно для фичи.
- Проверки должны опираться на устойчивые идентификаторы, предсказуемые тестовые данные и зафиксированные предусловия.
- Если для сценария нужны специальные флаги окружения, это должно быть согласовано с `DO` и отражено в `deployment-map.md`.

## Передача результата

- После выполнения задачи `QA` передаёт результат в виде краткого пакета: список проверок, способ запуска, окружение, наблюдаемый результат и известные ограничения.
- Если найдена проблема, `QA` передаёт её как воспроизводимый дефект с контекстом, а не как общий комментарий.
- Если проверка прошла, результат должен быть достаточен для следующего шага без дополнительного расспроса.

## Взаимодействие с `DO`

- `QA` передаёт `DO` не только факт прохождения e2e, но и все условия, которые нужны для запуска конвейера и дымовой проверки.
- `DO` использует результат `QA` как вход для развёртывания и post-deploy проверки, но не подменяет e2e своими операционными проверками.
- Если `DO` обнаруживает несоответствие окружения или проблемы конвейера, это возвращается в `QA` только как операционный блокер, а не как изменение сценария.

## Критерии допуска к ревью

- Проверки покрывают одну `FEATURE-*` и её заявленные сценарии.
- Результат проверки оформлен так, чтобы его можно было передать в `DO`.
- В задаче нет смешения e2e-ответственности с реализацией продукта или операциями развёртывания.
- Тестовые данные, предусловия и способ запуска описаны явно.
