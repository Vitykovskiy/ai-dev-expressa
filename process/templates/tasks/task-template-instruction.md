# Инструкция к шаблону задачи

## Назначение

Шаблон используется для постановки задач проекта. Карточка должна кратко фиксировать исполнителя, результат, границы работ, зону ответственности и обязательный маршрут чтения.

Карточка не должна превращаться в полный индекс документов проекта. Обязательные источники указываются в разделе `Маршрут чтения`; дополнительные материалы указываются в разделе `Справочные ссылки`.

## Язык и термины

- Заголовки шаблона пишутся по-русски.
- Смешанные служебные заголовки не используются.
- Английские термины допустимы только для технических имен, путей, команд, идентификаторов, названий файлов и закрепленных названий артефактов.
- Рекомендуемые русские аналоги берутся из `terms-map.md`.
- `Маршрут чтения` означает единственный обязательный маршрут чтения по умолчанию.
- `Справочные ссылки` означают необязательные ссылки. Исполнитель читает их только при явно зафиксированной необходимости расширить контекст.

## Основные поля

- `Идентификатор` задается в формате `<тип или сокращение роли>-<инкремент>`.
- Инкремент ведется отдельно по каждому типу карточки или роли и заполняется с лидирующими нулями: `001`, `002`, `003`.
- `Родительская задача` обязательна для дочерних задач и содержит идентификатор родительской `SPRINT-*` или `FEATURE-*` карточки.
- Для верхнеуровневой карточки спринта и аналитических задач без родителя используется значение `нет`.
- `Заголовок` формулируется кратко и в терминах результата.
- Для задач ручного тестирования заголовок начинается с `Ручное тестирование`.
- Для задач сквозного тестирования заголовок начинается с `E2E`.
- `Единица поставки` обязательна для `FEATURE-*` и для всех контурных задач под фичей.
- Для `SPRINT-*`, аналитических задач и задач без единицы поставки используется значение `n/a`.
- `Роль` выбирается из утвержденного списка ролей.
- `Контурная карта` указывает точечную architecture map или `n/a`.
- `Контурная карта` обязательна для `FE-*`, `BE-*`, `DO-*`, `QA-*` и `BUG-*`, если задача меняет или проверяет конкретный implementation contour.
- `Приоритет` выбирается из утвержденной шкалы приоритетов.
- `Статус` выбирается из утвержденного списка статусов.

## Цель

- `Цель` содержит короткую постановку задачи и ожидаемый проверяемый результат.
- `Цель` не дублирует подробные правила из feature package, контрактов, архитектурных карт или ролевых инструкций.
- Детали поведения, форматы ответов, ошибки и проверки доступа описываются в канонических документах и включаются в карточку только как краткая ссылка на результат.

## Границы задачи

- `Границы задачи` содержат только ограничения конкретной задачи.
- Общие правила из `process/workflow.md`, ролевых инструкций и других process-документов не дублируются.
- Для задач под `FEATURE-*` границы должны отделять текущую задачу от соседних фич и контуров.
- Для `BUG-*` границы должны фиксировать только исправляемый дефект и затронутый контур причины.

## Зона ответственности

- `Зона ответственности` является обязательным разделом для новых карточек.
- `Разрешенная зона правок` содержит только пути, artifact families, contours, design artifacts или проверки, которые исполнитель может менять или выполнять.
- `Запрещенная зона правок` содержит соседние контуры, upstream documents, downstream artifacts и типы изменений, которые не входят в задачу.
- Для задач без изменений файлов используется значение `только анализ без правок`.
- Для `FEATURE-*` разрешенная зона обычно ограничена feature package, системными артефактами и самой `FEATURE-*` карточкой.
- Для `DESIGN-*` разрешенная зона указывает named approved UI source или design artifact.
- Для `FE-*`, `BE-*`, `DO-*`, `QA-*` и `BUG-*` разрешенная зона должна совпадать с назначенным contour и контурной картой.
- Если executor не может определить допустимую зону правок из карточки, он фиксирует blocker до изменения файлов.

## Маршрут чтения

- `Маршрут чтения` является обязательным и достаточным маршрутом чтения по умолчанию.
- В `Маршрут чтения` включаются только источники, без которых задачу нельзя выполнить или проверить.
- Один источник добавляется один раз.
- Ссылки в `Маршруте чтения` располагаются от общего канонического источника к более точечному.
- Для задач под `FEATURE-*` первым системным источником обычно является package `index.md`.
- Package slices включаются в `Маршрут чтения` по role read routes из `index.md`.
- Package `test-scenarios.md` включается в `Маршрут чтения`, если задача является архитектурной передачей, ручным тестированием, сквозным тестированием или зависит от устойчивого идентификатора сценария.
- Контурная карта включается в `Маршрут чтения`, если задача меняет или проверяет факты конкретного контура.
- Файлы из `.references/` включаются в `Маршрут чтения` только как конкретные версионированные файлы, если они являются источником проверки соответствия интерфейса или поведения интерфейса.
- `DESIGN-*` задача, созданная из-за расхождения системного анализа и approved UI source, может указывать `не требуется; постановка задачи содержит полный контекст` вместо ссылочного маршрута чтения.
- Такая `DESIGN-*` задача должна быть самодостаточной и не должна требовать от дизайнера чтения `docs/system`, feature spec, contract, task tree, process docs или соседних task-card.
- Новые `DESIGN-*` задачи оформляются по `process/templates/tasks/design-task-template.md`.
- Прошлые карточки задач включаются в `Маршрут чтения` только для ретеста, последующей задачи по дефекту или восстановления результата, когда прошлый результат является фактом выполнения.
- Прошлые карточки задач не используются как образец формата, источник решений или замена канонического документа.
- Архивные карточки из `tasks/archive/` не включаются в `Маршрут чтения`, если текущая задача явно не требует исторический факт из архива.

## Ограничение объема маршрута чтения

- Обычные `FE-*`, `BE-*`, `DO-*` и `BUG-*` задачи содержат 3-7 источников в `Маршруте чтения`.
- `QA-*` задачи содержат 4-8 источников в `Маршруте чтения`.
- `AR-*` и `FEATURE-*` задачи содержат до 10 источников в `Маршруте чтения`, если задача действительно затрагивает несколько контуров.
- `SPRINT-*` карточки не задают исполнительный `Маршрут чтения`; они используют навигационные ссылки и список включенных фич.
- Если задаче нужен больший маршрут чтения, причина указывается в `Границах задачи`.

## Справочные ссылки

- `Справочные ссылки` содержат дополнительные материалы, которые помогают ориентироваться, но не являются обязательными для старта задачи.
- Справочная ссылка не становится обязательной, пока она не указана в `Маршруте чтения`.
- Если справочные материалы не нужны, указывается `не требуются`.
- Раздел не дублирует весь `Маршрут чтения`.

## Brief для `DESIGN-*`

- Новые `DESIGN-*` задачи создаются по специализированному шаблону `process/templates/tasks/design-task-template.md`.
- `DESIGN-*` содержит самодостаточную постановку для дизайнера без process jargon.
- В `Цель` фиксируется named approved UI source или design artifact, который должен быть обновлен.
- В `Границы задачи` фиксируется только пользовательская или экранная область изменения.
- В `Зона ответственности` фиксируется named design artifact как единственная разрешенная зона правок.
- В `Маршрут чтения` используется `не требуется; постановка задачи содержит полный контекст`, если brief полон.
- Содержимое brief описывает только что добавить, что изменить, что убрать, какие состояния или варианты проверить и какой результат считать готовым.
- Brief не требует чтения feature package, system docs, task tree, role prompts или `.references`, если нужные фрагменты уже включены в постановку.
- Brief для дизайнера должен быть пригоден для прямой передачи дизайнеру или Figma Make без дополнительных объяснений.
- Brief для дизайнера не содержит аналитическое обоснование, design readiness, architecture handoff, task-tree rationale, `scope`, `accepted scope`, `out-of-scope`, `future-flow` или feature-package terminology.
- Внутренние идентификаторы `FEATURE-*`, `SPRINT-*`, `AR-*`, `FE-*`, `BE-*`, `DO-*` и `QA-*` не используются внутри brief для дизайнера, если они не являются частью видимого пользовательского текста.
- Если дизайнеру нужно отделить один сценарий от другого, brief формулирует это как конкретное UI-изменение: отдельная группа, разделитель, отдельный диалог, disabled state, пояснение, подпись или проверяемый вариант.
- Аналитическое обоснование и границы фичи фиксируются в системных артефактах или аналитической task-card, а не в brief для дизайнера.

## Результат готовности

- `Результат готовности` фиксирует проверяемый результат, а не перечень действий.
- Для задач реализации результат готовности описывает наблюдаемое поведение, измененный контракт, готовую инфраструктуру или закрытый дефект.
- Для задач тестирования результат готовности описывает решение по приемке, пройденные сценарии и созданные дефекты или их отсутствие.
- Для аналитических и архитектурных задач результат готовности описывает готовый канонический артефакт или передачу в следующий контур.

## Проверки

- `Проверки` фиксируют конкретные команды, ручные сценарии, дымовые проверки или приемочные действия.
- Для `FE-*`, `BE-*` и `DO-*` задач указываются только проверки, относящиеся к затронутому контуру.
- Для ручного тестирования указываются идентификатор сценария, маршрут проверки и правило фиксации дефектов.
- Для сквозного тестирования указываются команда запуска, окружение, идентификатор сценария, файл теста, название теста и обязательные утверждения.
- Если проверка невозможна до внешнего события, причина фиксируется прямо в этом разделе.

## Результат выполнения

- `Результат выполнения` заполняется только после выполнения задачи.
- Новая задача содержит значение `не заполнено`.
- В выполненной задаче раздел содержит дату, краткий итог, результат проверок и ссылки на созданные дефекты или последующие задачи.
- Отчет не должен повторять постановку задачи и весь маршрут чтения.

## Типы карточек и сокращения ролей для идентификатора

- `SPRINT` — координационная карточка спринта.
- `FEATURE` — карточка единицы поставки.
- `AR` — Архитектор.
- `BA` — Бизнес аналитик.
- `SA` — Системный аналитик.
- `DO` — Девопс.
- `FE` — Фронтенд.
- `BE` — Бэкенд.
- `QA` — Тестирование.
- `BUG` — воспроизводимый дефект с меткой контура причины.
- `DESIGN` — дизайнерская задача для правки approved UI source или макета.

## Допустимые роли

- Архитектор
- Разработка
- Бизнес аналитик
- Системный аналитик
- Девопс
- Дизайнер
- Фронтенд
- Бэкенд
- Тестирование

## Допустимые статусы

- В работе

- В тестировании
- Выполнена

## Градация приоритетов

| Приоритет     | Когда использовать                                                            | Что означает для очереди                                                 |
| ------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `Критический` | Запуск продукта или продолжение потока поставки невозможны без этой задачи    | Брать в работу раньше всех остальных задач со статусом `Готова к работе` |
| `Высокий`     | Задача нужна для основного пути v1 или разблокирует несколько следующих задач | Брать в первой волне после критических                                   |
| `Средний`     | Задача завершает подтвержденную область работ, но не мешает старту ядра       | Брать после задач, открывающих основной поток                            |
| `Низкий`      | Локальное улучшение или изолированный результат без блокировки других задач   | Брать в последнюю очередь                                                |

## Модель потока поставки

- Общая модель потока поставки описана в `process/workflow.md`.
- Эта инструкция фиксирует только правила заполнения карточки по `process/templates/tasks/task-template.md`.
