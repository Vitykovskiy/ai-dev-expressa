# Кодстайл и quality gates

## Общие правила

- Писать на `TypeScript` в строгом режиме.
- Имена модулей, пакетов и директорий должны совпадать с `application-map.md`.
- Не смешивать `frontend`, `backend`, `bot` и `devops` логику в одной child-задаче без отдельного архитектурного решения.
- Любое изменение структуры, entrypoints, shared contracts, env/config или deployment path требует обновления `application-map.md`.
- Если корневая навигация по репозиторию изменилась, обновлять `README.md`.

## Frontend

- Использовать `Vue 3` Composition API и `Vuetify`.
- Во `Vue` single-file components секции располагать в порядке: `template`, затем `script`, затем `style`.
- UI-логика с ветвлением должна жить в composables, local reactive state modules или adapter-слое, а не быть размазана по шаблонам.
- Не дублировать API-контракты вручную между приложениями; shared types выносить в `packages/shared-types/`.

## Backend

- Доменная и прикладная логика должна быть отделена от transport/integration слоя.
- Сервисы, валидаторы, мапперы и правила переходов состояния должны быть тестируемыми без запуска внешних интеграций.
- Изменения в контрактах должны синхронно отражаться в shared types и в карте приложения.

## DevOps

- Pipeline должен быть воспроизводимым из репозитория и описан в `deployment-map.md`.
- Любой новый secret, env var, deploy target или smoke-check должен быть отражен в документации до завершения задачи.

## Unit tests и проверки

- `FE`: обязательны unit tests для state modules, composables, форматтеров, адаптеров API, валидаторов и любой UI-логики с ветвлением.
- `BE`: обязательны unit tests для доменных правил, сервисов, валидаторов, мапперов, вычислений и правил переходов состояния.
- `DO`: вместо unit tests обязательны pipeline-validation, deploy smoke-check и документированный rollback/restore path.
- Исключение из unit tests допустимо только для тривиальной разметки или тривиальной конфигурации и должно быть явно указано в задаче или PR.

## Review gates

- Задача считается готовой к ревью только если обновлены релевантные документы из `docs/architecture/`.
- Ревьюер должен видеть один завершенный outcome в одном контуре.
- Если change set нельзя проверить отдельной командой, unit test или smoke-сценарием, задача декомпозирована недостаточно.
