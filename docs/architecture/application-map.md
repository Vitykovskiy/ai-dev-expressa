# Карта приложения

## Назначение

Этот документ помогает быстро понять, где находится каждый контур приложения, как он запускается, чем он зависит от соседних модулей и какие изменения требуют обновления документации.

## Текущее состояние

- В репозитории существуют два реальных исполняемых контура: `apps/server` и `apps/backoffice-web`.
- В `apps/server` реализован серверный срез `FEATURE-001`: идемпотентный bootstrap главного `administrator`, bootstrap доступа во внутренний административный контур, in-memory сессии доступа и серверные guard-правила.
- В `apps/backoffice-web` реализован клиентский интеграционный слой `FEATURE-001`: pre-bootstrap проверка Telegram-контекста, test environment-поведение при `VITE_DISABLE_TG_AUTH=true`, HTTP bootstrap доступа, восстановление сессии по `accessToken`, хранение пользовательского контекста, композиционная функция подготовки shell-состояния, серверно-управляемая навигация вкладок, route guards и экран отказа в доступе.
- В `packages/shared-types` добавлен общий пакет деклараций типов для bootstrap доступа во внутренний административный контур.
- Для `FEATURE-002` зафиксирована архитектурная рамка управления каталогом меню: фича расширяет существующие `apps/server`, `apps/backoffice-web` и `packages/shared-types` без новых исполняемых контуров, обязательных переменных окружения и отдельного маршрута развёртывания.
- В `e2e/feature-001` добавлен Playwright-набор `QA-001`, который поднимает `apps/server` и `apps/backoffice-web` для сквозной проверки Telegram-входа `administrator`.
- В `infra/docker` добавлен контейнерный маршрут `FEATURE-001`: compose-файл, шаблоны окружения и упаковка `apps/server` и `apps/backoffice-web` для VPS.
- В `infra/scripts` добавлены `deploy-feature001.sh` и `smoke-backoffice-access.mjs` для воспроизводимого развёртывания и дымовой проверки.
- В `.github/workflows` добавлены workflow `ci.yml` и `deploy-feature001.yml`, которые валидируют маршрут поставки и выполняют развёртывание на VPS.
- Каталог `apps/customer-web` пока не создан и остаётся целевым путём следующих задач.

## Архитектурная рамка `FEATURE-001`

### Границы ответственности

| Контур | Ответственность в `FEATURE-001` | Не входит в контур |
| --- | --- | --- |
| `apps/server` | Идемпотентно создаёт главного `administrator` из `ADMIN_TELEGRAM_ID` при старте; принимает и валидирует Telegram-контекст служебного бота; вычисляет пользовательский контекст доступа; применяет серверные guard-правила по каналу входа, роли и блокировке | Рендеринг вкладок, маршрутизация клиентской части, визуальный экран отказа |
| `apps/backoffice-web` | Читает Telegram-контекст из веб-приложения; выполняет pre-bootstrap проверку канала входа; в test environment переключается на test-mode payload; хранит `accessToken` и контекст текущего пользователя; восстанавливает сессию через `GET /api/backoffice/access/me`; строит навигационный каркас и route guards по `availableTabs`; рендерит экран отказа в доступе для недопущенных сценариев | Проверка подписи Telegram, bootstrap главного `administrator`, окончательное решение о доступе в серверную часть |
| `packages/shared-types` | Хранит общие типы для bootstrap доступа: DTO запроса и ответа, перечисления ролей, причин отказа и списка допустимых вкладок | Доменные сервисы, HTTP-клиент, UI-логика и guard-реализация |

### Реализованный серверный контракт `FEATURE-001`

| Метод и путь | Назначение | Основной результат |
| --- | --- | --- |
| `POST /api/backoffice/access/bootstrap` | Bootstrap доступа во внутренний административный контур по Telegram-контексту или test-mode payload | Возвращает пользователя, роли, доступные вкладки, канал входа и `accessToken` |
| `GET /api/backoffice/access/me` | Чтение текущего серверного контекста доступа по `Bearer`-токену | Возвращает нормализованный контекст текущей сессии |
| `GET /api/backoffice/admin/ping` | Дымовая и модульная проверка guard-правила `administrator` | Возвращает `200` только для сессии с ролью `administrator` |

### Реализованные серверные оговорки

- До добавления постоянного слоя хранения данных `apps/server` использует in-memory адаптер `UserRepositoryPort`, чтобы изолировать доменные правила доступа от будущей интеграции с PostgreSQL и `TypeORM`.
- Сессии доступа к внутреннему административному контуру также хранятся в памяти процесса и предназначены для локальной разработки, модульных тестов и первой поставки `BE-001`.
- Валидация Telegram-контекста опирается на `TG_BACKOFFICE_BOT_TOKEN`, поэтому сервер принимает только контекст служебного бота.

### Поток входа `administrator`

1. При старте `apps/server` выполняется идемпотентный bootstrap главного `administrator` из `ADMIN_TELEGRAM_ID`.
2. Служебный бот открывает `apps/backoffice-web` внутри Telegram и передаёт веб-приложению Telegram-контекст.
3. `apps/backoffice-web` отправляет в `apps/server` один запрос bootstrap доступа для backoffice: в рабочем режиме с Telegram-контекстом, в test environment с явным test-mode payload.
4. `apps/server` валидирует канал входа, определяет `telegramId`, находит пользователя, проверяет статус блокировки и набор ролей.
5. `apps/server` возвращает нормализованный контекст доступа: пользователь, роли, доступные вкладки, признак test-mode и причину отказа при недопуске.
6. `apps/backoffice-web` строит навигацию и route guards только из ответа серверной части и не вычисляет права доступа самостоятельно.
7. При прямом рабочем переходе по недоступному URL клиентская часть останавливает маршрут через route guard и показывает экран отказа в доступе вместо обхода ограничений.

### Правила test environment

- `DISABLE_TG_AUTH=true` меняет только способ подтверждения канала входа, но не отменяет проверки роли, блокировки и принадлежности к backoffice.
- В test environment `apps/backoffice-web` и `apps/server` используют тот же контракт bootstrap доступа, что и в рабочем режиме, но вместо Telegram-контекста передаётся test-mode payload.
- В рабочем режиме `apps/server` обязан отклонять любые backoffice-запросы без валидного Telegram-контекста служебного бота.
- Логика test environment не должна проникать в модули меню, пользователей, настроек и заказов; исключение локализуется в слое входа и guard-правилах `FEATURE-001`.

### Guard-правила и точки отказа

| Слой | Обязательное правило | Результат отказа |
| --- | --- | --- |
| Клиентская часть, pre-bootstrap | Если Telegram-контекст отсутствует и test environment не активирован, `apps/backoffice-web` не открывает рабочую навигацию | Экран отказа без загрузки вкладок и данных |
| Серверная часть, bootstrap доступа | Если нет валидного Telegram-контекста служебного бота и `DISABLE_TG_AUTH` не включён, `apps/server` не создаёт пользовательский контекст backoffice | Ответ `401/403` с причиной отказа |
| Серверная часть, guard роли | Пользователь без роли `barista` или `administrator`, а также пользователь со статусом `blocked`, не допускается в backoffice | Ответ `403` с причиной отказа |
| Клиентская часть, guard маршрута | Недоступные по роли вкладки не рендерятся; прямой переход по маршруту `menu/users/settings` без роли `administrator` показывает `403`-экран | Пользователь остаётся в разрешённом контуре без обхода по URL |

## Архитектурная рамка `FEATURE-002`

### Границы ответственности

| Контур | Ответственность в `FEATURE-002` | Не входит в контур |
| --- | --- | --- |
| `apps/server` | Предоставляет `administrator` чтение и сохранение структурного снимка каталога меню; валидирует состав категорий, товаров, цен, размеров напитков `S/M/L`, групп дополнительных опций и правил выбора; сохраняет структуру каталога отдельно от оперативной доступности | Маршрутизация вкладки `menu`, локальное состояние редакторов, клиентские формы и визуальные ошибки |
| `apps/backoffice-web` | Загружает структурный снимок каталога, хранит черновое состояние вкладки `menu`, строит подпотоки `menu.menu_categories`, `menu.menu_products`, `menu.menu_product_detail`, `menu.addon_group_detail`, показывает серверные ошибки и валидации форм | Самостоятельное вычисление канонической структуры каталога, сохранение без серверной валидации, изменение оперативной доступности товара и дополнительной опции |
| `packages/shared-types` | Хранит общие DTO структурного снимка каталога, перечисления типа товара, размеров напитков и режима выбора в группе дополнительных опций, а также коды бизнес-ошибок сохранения | HTTP-клиент, доменные сервисы, хранилища состояния и шаблоны UI |

### Целевой серверный контракт `FEATURE-002`

| Метод и путь | Назначение | Основной результат |
| --- | --- | --- |
| `GET /api/backoffice/menu/catalog` | Чтение структурного снимка каталога меню для административного редактирования | Возвращает категории, товары, цены, размеры напитков, группы дополнительных опций и их связи в одном согласованном снимке |
| `PUT /api/backoffice/menu/catalog` | Сохранение структурного снимка каталога меню после серверной валидации | Возвращает нормализованный снимок каталога после сохранения |

### Контрактные оговорки `FEATURE-002`

- `FEATURE-002` использует существующий контур доступа `FEATURE-001`: оба запроса защищаются тем же `accessToken` и guard-правилом `administrator`.
- Сохранение каталога выполняется как атомарное сохранение одного структурного снимка, а не как набор независимых операций редактирования. Это нужно, чтобы серверная часть могла валидировать размеры напитков, связи категорий с группами дополнительных опций и целостность каталога за один проход.
- Структурный снимок каталога не должен включать временную доступность товаров и дополнительных опций как редактируемое поле `administrator`. Оперативная доступность остаётся отдельным сценарием `barista` и не должна теряться при административном сохранении каталога.
- Если `BE-002` начнёт с in-memory реализации, порт хранения всё равно должен разделять структурный каталог и оперативную доступность, чтобы следующая интеграция с `PostgreSQL` не смешала эти состояния.
- Минимальный набор бизнес-ошибок `FEATURE-002` остаётся общим для серверной части и клиентской части: `administrator-role-required`, `invalid-drink-size-model`, `invalid-option-group-rule`. Любое расширение перечня сначала фиксируется в `packages/shared-types`.

### Маршруты и состояние вкладки `menu`

- Верхнеуровневая вкладка `menu`, добавленная в `FEATURE-001`, остаётся единственной точкой входа в административное управление каталогом и не порождает новую вкладку верхнего уровня.
- Внутри `apps/backoffice-web` подпотоки `FEATURE-002` резервируются под имена маршрутов из `backoffice-ui-binding.md`: `menu.menu_categories`, `menu.menu_products`, `menu.menu_product_detail`, `menu.addon_group_detail`.
- Клиентская часть должна вести единый источник истины по снимку каталога в хранилище состояния вкладки `menu`, а не распылять черновые данные по независимым компонентам и маршрутам.
- `FE-004` отвечает за чтение снимка, навигацию подпотоков и базовую оркестрацию состояния. `FE-005` опирается на этот слой и добавляет редакторы, валидации формы и обработку серверных ошибок без дублирования канонических правил.

### Ограничения модели `FEATURE-002`

- Товар типа `drink` обязан содержать полный набор цен по размерам `S`, `M`, `L`; товар без обязательного выбора размера использует одну базовую цену.
- Группа дополнительных опций назначается на категорию и наследуется всеми товарами этой категории; в рамках `FEATURE-002` нет отдельного механизма переопределения состава групп на уровне конкретного товара.
- Режим выбора группы дополнительных опций должен быть явным: либо множественный выбор, либо взаимоисключающий выбор одного варианта.
- Клиентская часть может выполнять только валидации обязательности полей и целостности формы, но серверная часть остаётся единственным источником истины по модели каталога.

## Архитектурные оговорки

- Каталоги `apps/customer-bot` и `apps/backoffice-bot` не планируются: оба Telegram-контура реализуются как NestJS-модули внутри `apps/server`.
- Пакет `packages/ui` не планируется: customer UI и backoffice UI развиваются независимо в своих приложениях.
- После появления реальных каталогов `apps/server`, `apps/backoffice-web` и `packages/shared-types` архитектурная рамка `FEATURE-001` остаётся обязательной для следующих задач `FE-002`, `FE-003`, `DO-001` и `QA-001`.
- Для `FEATURE-002` отдельная задача `DO-*` не выделяется, пока фича укладывается в существующие исполняемые контуры, переменные окружения и маршрут развёртывания `FEATURE-001`.

## Точки входа и маршруты

- `apps/customer-web`: точка входа в клиентское Telegram веб-приложение.
- `apps/backoffice-web`: точка входа в backoffice Telegram веб-приложение; прямой рабочий доступ по URL вне Telegram допускается только при `DISABLE_TG_AUTH=true`.
- `apps/server`: единая точка входа NestJS для HTTP API, bootstrap доступа в backoffice, webhook/polling Telegram-ботов и фоновых процессов уведомлений.
- `packages/shared-types`: единая точка публикации общих типов контрактов между `apps/server` и веб-приложениями.
- `FEATURE-002`: внутри существующей точки входа `apps/server` резервируются `GET /api/backoffice/menu/catalog` и `PUT /api/backoffice/menu/catalog` для чтения и сохранения структурного снимка каталога меню.
- `FEATURE-002`: внутри `apps/backoffice-web` вкладка `menu` разветвляется на подпути категорий, товаров, карточки товара и карточки группы дополнительных опций; все подпути остаются внутри уже существующего `administrator`-guard.

## Где запускать и проверять

- Установка зависимостей: `npm install`
- Сборка серверной части: `npm run build --workspace @expressa/server`
- Сборка `apps/backoffice-web`: `npm run build --workspace @expressa/backoffice-web`
- Модульные и HTTP-тесты серверной части: `npm run test --workspace @expressa/server`
- Модульные тесты `apps/backoffice-web`: `npm run test --workspace @expressa/backoffice-web`
- Проверка контейнерного маршрута `FEATURE-001`: `npm run deploy:feature001:config`
- Дымовая проверка `FEATURE-001`: `npm run smoke:backoffice-access`
- E2E-проверка `FEATURE-001`: `npm run test:e2e`
- Локальный запуск серверной части: `npm run start:dev --workspace @expressa/server`
- Локальный запуск `apps/backoffice-web`: `npm run dev --workspace @expressa/backoffice-web`
- Для ручной рабочей проверки `FEATURE-001` серверу нужны `ADMIN_TELEGRAM_ID`, `TG_BACKOFFICE_BOT_TOKEN` и при необходимости `PORT`; базовый шаблон лежит в `apps/server/.env.example`.
- Для локального каркаса `apps/backoffice-web` базовый шаблон переменных окружения лежит в `apps/backoffice-web/.env.example`; на этапе `FE-001` обязательны `VITE_APP_TITLE` и `VITE_API_BASE_URL`.
- Для контейнерного маршрута `FEATURE-001` шаблоны окружения лежат в `infra/docker/.env.example` и `infra/docker/.env.server.example`, а compose-файл — в `infra/docker/compose.feature001.yml`.
- На этапе `FE-002` клиентская часть дополнительно хранит `accessToken` в `localStorage`, при старте сначала пытается восстановить сессию через `GET /api/backoffice/access/me`, а затем при необходимости выполняет `POST /api/backoffice/access/bootstrap`.
- На этапе `FE-003` клиентская часть дополняет этот поток pre-bootstrap проверкой Telegram-контекста и, если `VITE_DISABLE_TG_AUTH=true`, переключением на test-mode payload c `VITE_TEST_TELEGRAM_ID`.
- Для локального test environment в `apps/backoffice-web/.env.example` дополнительно фиксируются `VITE_DISABLE_TG_AUTH` и `VITE_TEST_TELEGRAM_ID`; эти переменные не заменяют серверные guard-правила и не дают доступ без серверного решения.
- Для `FEATURE-001` локальная и конвейерная проверка отдельно подтверждают рабочий вход `administrator`, отказ при прямом рабочем доступе без Telegram и выделенный test environment-прогон через `SMOKE_MODE=test`.
- Сквозные проверки `FEATURE-001` расположены в `e2e/feature-001` и поднимают `apps/server` и `apps/backoffice-web` с тестовыми env-параметрами через Playwright `webServer`.
- VPS-маршрут `FEATURE-001` выполняется через `.github/workflows/deploy-feature001.yml`, который синхронизирует репозиторий на сервер, рендерит `infra/docker/.env*`, запускает `infra/scripts/deploy-feature001.sh` и завершает выпуск post-deploy дымовой проверкой.
- `FEATURE-002` использует те же команды сборки, модульных тестов и e2e-проверок, что уже зафиксированы для `apps/server`, `apps/backoffice-web` и `Playwright`; отдельные команды, env-файлы и workflow для архитектурной рамки не требуются.
- Маршруты развёртывания для конкретных окружений читаются из `deployment-map.md`.

## Когда обновлять карту

- Появился новый каталог верхнего уровня или новый исполняемый контур.
- Изменилась точка входа, способ запуска, путь к тестам или путь к артефакту развёртывания.
- Появился новый общий пакет, новое внешнее ребро зависимости или новый интеграционный модуль.
- Изменились переменные окружения, которые важны для запуска, тестирования или деплоя.
