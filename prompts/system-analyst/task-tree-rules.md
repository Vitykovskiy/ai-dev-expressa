# Правила дерева задач для спринта и фичи

## Назначение

Эта инструкция определяет, как системный аналитик должен формировать дерево задач для одного спринта при подготовке задач для разработки.

## Правила

- Одна верхнеуровневая задача `SPRINT-*` должна представлять ровно один спринт и ничего за его пределами.
- `FEATURE-*` является единственной карточкой единицы поставки: одна фича — это один завершенный, работоспособный, тестируемый и демонстрируемый результат для customer или stakeholder.
- Внутренний технический prerequisite, аналитический системный срез или отдельная capability не являются самостоятельной фичей, если их нельзя принять как отдельный законченный результат.
- Системный аналитик создает или обновляет только карточку `SPRINT-*`.
- Системный аналитик не должен заранее создавать `FEATURE-*` и не должен заранее создавать дочерние задачи архитектора, backend, frontend, devops или QA.
- Архитектор сначала декомпозирует `SPRINT-*` на набор `FEATURE-*`.
- После этого архитектор декомпозирует только одну выбранную `FEATURE-*` за раз на задачи `AR/FE/BE/DO/QA-*`.
- `QA-*` обязательна для каждой фичи и должна включать e2e-тесты этой фичи.
- `DO-*` создается только если по фиче действительно меняются VPS-окружение, staging или production-развёртывание, либо GitHub Actions.
- Пока хотя бы одна `FEATURE-*` внутри спринта не завершена, `SPRINT-*` остается в статусе `В работе`.
- `FEATURE-*` переходит в статус `Ожидает тестирования` только после того, как завершены все обязательные контурные задачи этой фичи.
- Тестировщик берет `FEATURE-*` только после того, как она перешла в статус `Ожидает тестирования`.
- `FEATURE-*` закрывается только после успешного прохождения e2e и проверки собранного результата.
- `SPRINT-*` закрывается только после того, как завершены все включенные в него `FEATURE-*`.
- Нельзя объединять несколько законченных фич в одну карточку `FEATURE-*`, даже если они относятся к одному бизнес-сценарию или были выделены в рамках одной аналитической сессии.

## Примечания

- Аналитическая декомпозиция системы на меню, слоты, уведомления, доступ и другие системные срезы не равна декомпозиции спринта на фичи.
- `SPRINT-*` является координационной границей для группы фич, а `FEATURE-*` — координационной границей для одной единицы поставки.
- E2e-тесты живут в `QA-*` для конкретной фичи и не заменяют собой саму карточку `FEATURE-*`.
