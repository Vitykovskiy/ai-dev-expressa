# Карточка задачи

## Карточка задачи

- Идентификатор: `DESIGN-001`
- Родительская задача: `FEATURE-004`
- Заголовок: `Исправить макет назначения роли на экране Пользователи`
- Единица поставки: `FEATURE-004`
- Роль: `Дизайнер`
- Приоритет: `Критический`
- Статус: `Готова к работе`

## Цель

`Подготовить исправленный approved UI source для экрана Пользователи, чтобы сценарий назначения роли существующему пользователю был однозначен и не смешивался со сценарием создания нового пользователя.`

## Полное описание для дизайнера

В фиче `FEATURE-004` administrator должен просматривать список пользователей и назначать существующему пользователю роль `barista` или `administrator`.

Текущий системный контракт фичи работает так:

- Система получает список существующих пользователей.
- Administrator выбирает существующего пользователя.
- Система отправляет операцию назначения роли для выбранного пользователя.
- Операция назначения роли использует существующий `userId`.
- Операция назначения роли не создает нового пользователя.
- Операция назначения роли не изменяет роль `customer`.
- Операция назначения роли не блокирует и не разблокирует пользователя.

Проблема текущего макета:

- На экране есть кнопка `Добавить пользователя`.
- Диалог называется `Новый пользователь`.
- В диалоге есть поля `Имя`, `Telegram Username` и `Роль`.
- Кнопка подтверждения называется `Добавить пользователя`.
- Такой интерфейс выглядит как создание нового пользователя.
- Для `FEATURE-004` нужен сценарий назначения роли уже существующему пользователю.
- Сейчас в макете неясно, как выбирается существующий пользователь для назначения роли.
- Сейчас в макете не раскрыто, что находится в меню действий строки пользователя.
- Из-за этого следующий исполнитель должен сам выбрать UX-решение, а это недопустимо.

## Требуемое решение в макете

Дизайнер должен явно развести два сценария:

- создание нового пользователя;
- назначение роли существующему пользователю.

Для `FEATURE-004` нужно покрыть именно назначение роли существующему пользователю.

Макет должен явно показать:

- где administrator видит список существующих пользователей;
- как administrator выбирает конкретного пользователя;
- как открывается действие назначения роли;
- какие элементы доступны в меню действий строки пользователя;
- как называется действие назначения роли;
- какой диалог или панель открывается после выбора действия;
- какие данные выбранного пользователя отображаются в форме;
- как выбирается новая роль;
- как выглядит подтверждение операции;
- как выглядит отмена операции;
- как выглядит недоступное состояние, если пользователю нельзя назначить роль;
- как выглядит состояние ошибки;
- как выглядит успешное завершение операции;
- как этот же сценарий работает на desktop;
- как этот же сценарий работает на mobile.

## Обязательные UX-решения

Макет должен зафиксировать один основной entrypoint назначения роли.

Допустимые варианты:

- действие `Назначить роль` в меню строки пользователя;
- отдельная кнопка действия в строке пользователя;
- другой явно показанный способ выбора существующего пользователя.

Если в макете остается top-level кнопка `Добавить пользователя`, она должна относиться только к сценарию создания пользователя или должна быть переименована так, чтобы не конфликтовать с назначением роли существующему пользователю.

Если создание пользователя не входит в текущий сценарий, макет должен не заставлять пользователя проходить через форму `Новый пользователь` для назначения роли.

## Тексты, которые нужно пересмотреть

Текущие подписи конфликтуют со сценарием назначения роли:

- `Добавить пользователя`
- `Новый пользователь`
- `Добавьте нового пользователя в систему`
- `Добавить пользователя` на кнопке подтверждения

Для операции назначения роли нужны подписи, из которых понятно, что administrator меняет роль выбранного существующего пользователя, а не создает нового.

## Ограничения

- Дизайнер не меняет системный контракт фичи.
- Дизайнер не добавляет блокировку пользователя в границы `FEATURE-004`.
- Дизайнер не добавляет разблокировку пользователя в границы `FEATURE-004`.
- Дизайнер не добавляет изменение роли `customer` в границы `FEATURE-004`.
- Дизайнер не обязан проектировать backend/API.
- Дизайнер не обязан читать проектные документы.
- Вся необходимая постановка содержится в этой карточке.

## Результат готовности

`Approved UI source экрана Пользователи однозначно показывает сценарий назначения роли существующему пользователю и больше не смешивает его с созданием нового пользователя.`

## Проверки

- В макете есть явный способ выбрать существующего пользователя для назначения роли.
- В макете есть явное действие назначения роли.
- В макете раскрыто меню действий строки пользователя, если назначение роли открывается из строки.
- Диалог или панель назначения роли не выглядит как создание нового пользователя.
- Подписи действия, заголовка и подтверждения соответствуют назначению роли существующему пользователю.
- Desktop и mobile варианты покрывают один и тот же сценарий.
- Сценарии блокировки, разблокировки и изменения роли `customer` не включены в `FEATURE-004`.
