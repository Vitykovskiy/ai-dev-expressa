# Кодстайл и критерии качества

## Общие правила

- Писать на `TypeScript` в строгом режиме.
- Имена модулей, пакетов и директорий должны совпадать с `application-map.md`.
- Не смешивать `frontend`, `backend`, `bot`, `devops` и `qa` логику в одной дочерней задаче без отдельного архитектурного решения.
- Любое изменение структуры, точек входа, общих контрактов, окружения и конфигурации или маршрута развёртывания требует обновления `application-map.md`.
- Если корневая навигация по репозиторию изменилась, обновлять `README.md`.

## Frontend

- Использовать `Vue 3` Composition API и `Vuetify`.
- UI-логика с ветвлением должна жить в composables, stores или слое адаптеров, а не быть размазана по шаблонам.
- Не дублировать API-контракты вручную между приложениями; общие типы выносить в `packages/shared-types/`.

## Backend

- Доменная и прикладная логика должна быть отделена от слоя транспорта и интеграций.
- Сервисы, валидаторы, мапперы и правила переходов состояния должны быть тестируемыми без запуска внешних интеграций.
- Изменения в контрактах должны синхронно отражаться в общих типах и в карте приложения.

## DevOps

- Pipeline должен быть воспроизводимым из репозитория и описан в `deployment-map.md`.
- `DO` отвечает только за VPS-окружение, GitHub Actions, маршруты развёртывания, дымовые проверки и порядок восстановления.
- `DO` не создает e2e-тесты и не владеет прикладным тестовым сценарием фичи.
- Любой новый секрет, переменная окружения, цель развёртывания или smoke-check должен быть отражен в документации до завершения задачи.

## QA

- `QA` создает и поддерживает e2e-тесты для каждой `FEATURE-*`.
- E2e-тесты должны проверять законченную пользовательскую или операционную функциональность, а не внутренний технический срез.
- Если для фичи нужен запуск e2e в конвейере или после развёртывания, `QA` определяет набор тестов, а `DO` обеспечивает их запуск в нужной среде.

## Модульные тесты и проверки

- `FE`: обязательны модульные тесты для stores, composables, форматтеров, адаптеров API, валидаторов и любой UI-логики с ветвлением.
- `BE`: обязательны модульные тесты для доменных правил, сервисов, валидаторов, мапперов, вычислений и правил переходов состояния.
- `DO`: вместо модульных тестов обязательны pipeline-validation, дымовая проверка после развёртывания и документированный откат или порядок восстановления.
- `QA`: обязательны e2e-тесты для каждой `FEATURE-*` и подтверждение их прохождения на собранной фиче.
- Исключение из модульных тестов допустимо только для тривиальной разметки или тривиальной конфигурации и должно быть явно указано в задаче или PR.

## Критерии допуска к ревью

- Задача считается готовой к ревью только если обновлены релевантные документы из `docs/architecture/`.
- Ревьюер должен видеть один завершенный результат в одном контуре.
- Если набор изменений нельзя проверить отдельной командой, модульным тестом или smoke-сценарием, задача декомпозирована недостаточно.
