# Карта серверной части

## Назначение

Эта карта фиксирует фактическое устройство `apps/server`: ответственность серверной части, модули, точки входа, зависимости, команды запуска и feature-срезы. Её читают только задачи, которые меняют серверную часть или серверные контракты.

## Ответственность и зависимости

| Контур | Путь | Ответственность | Основные зависимости |
| --- | --- | --- | --- |
| Серверная часть | `apps/server` | NestJS-приложение с общим каталогом `src/common`, модулями `src/modules/access` и `src/modules/menu`, HTTP API, guard-правилами, доменными сервисами и in-memory адаптерами до появления постоянного слоя хранения данных | `packages/shared-types`, `NestJS`, `Telegram Bot API` |

## Модули и точки входа

- `apps/server/src/main.ts` — точка входа NestJS-приложения.
- `apps/server/src/app.module.ts` — корневой модуль приложения.
- `apps/server/src/common` — общий серверный каркас, включая фильтр прикладных ошибок.
- `apps/server/src/modules/access` — модуль доступа `FEATURE-001` со слоями `application`, `domain`, `transport`, `infrastructure`.
- `apps/server/src/modules/menu` — модуль управления каталогом меню `FEATURE-002` со слоями `application`, `domain`, `transport`, `infrastructure`.
- `apps/server/test` — HTTP- и модульные тесты серверной части.
- `apps/server/.env.example` — пример переменных окружения серверной части.
- Каталоги `apps/customer-bot` и `apps/backoffice-bot` не планируются: оба Telegram-контура реализуются как NestJS-модули внутри `apps/server`.

## Feature-срезы

### `FEATURE-001`: доступ во внутренний административный контур

- При старте серверная часть идемпотентно создаёт главного `administrator` из `ADMIN_TELEGRAM_ID`.
- `POST /api/backoffice/access/bootstrap` принимает и валидирует Telegram-контекст служебного бота или test-mode payload, вычисляет пользовательский контекст доступа и возвращает пользователя, роли, доступные вкладки, канал входа и `accessToken`.
- `GET /api/backoffice/access/me` читает текущий серверный контекст доступа по `Bearer`-токену.
- `GET /api/backoffice/admin/ping` используется для дымовой и модульной проверки guard-правила `administrator`.
- Guard-правила проверяют канал входа, роль, блокировку пользователя и принадлежность к внутреннему административному контуру.
- До добавления постоянного слоя хранения данных используются in-memory адаптеры `UserRepositoryPort` и хранилище сессий доступа.
- `DISABLE_TG_AUTH=true` меняет только способ подтверждения канала входа в test environment и не отменяет проверки роли, блокировки и принадлежности к backoffice.
- Для ручной рабочей проверки серверной части нужны `ADMIN_TELEGRAM_ID`, `TG_BACKOFFICE_BOT_TOKEN` и при необходимости `PORT`; базовый шаблон лежит в `apps/server/.env.example`.

### `FEATURE-002`: управление каталогом меню

- `GET /api/backoffice/menu/catalog` возвращает согласованный структурный снимок каталога меню для административного редактирования.
- `PUT /api/backoffice/menu/catalog` сохраняет структурный снимок после серверной валидации и возвращает нормализованный снимок.
- Запросы используют существующий контур доступа `FEATURE-001`, `accessToken` и guard-правило `administrator`.
- Серверная часть валидирует состав категорий, товаров, цен, размеров напитков `S/M/L`, групп дополнительных опций и правил выбора.
- Сохранение каталога выполняется атомарно одним структурным снимком, чтобы серверная часть оставалась источником истины по модели каталога.
- Структурный снимок не включает оперативную доступность товаров и дополнительных опций как редактируемое поле `administrator`; доступность остаётся отдельным сценарием `barista`.
- Порт хранения разделяет структурный каталог и оперативную доступность, чтобы будущая интеграция с `PostgreSQL` не смешала эти состояния.
- Товар типа `drink` обязан содержать полный набор цен по размерам `S`, `M`, `L`; товар без обязательного выбора размера использует одну базовую цену.
- Группа дополнительных опций назначается на категорию и наследуется всеми товарами этой категории; отдельного переопределения состава групп на уровне конкретного товара в `FEATURE-002` нет.
- Режим выбора группы дополнительных опций должен быть явным: множественный выбор или взаимоисключающий выбор одного варианта.

## Команды запуска и проверки

- Установка зависимостей: `npm install`.
- Локальный запуск серверной части: `npm run start:dev --workspace @expressa/server`.
- Сборка серверной части: `npm run build --workspace @expressa/server`.
- Модульные и HTTP-тесты серверной части: `npm run test --workspace @expressa/server`.

## Когда обновлять карту

- Изменилась структура `apps/server`, точка входа, модульная граница или слой.
- Изменился HTTP-контракт, guard-правило, общий серверный порт или внешний адаптер.
- Появилась новая серверная зависимость или переменная окружения.
- Изменился способ запуска, сборки или тестирования серверной части.
