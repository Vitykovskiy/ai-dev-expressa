# Domain Model: Ordering And Pickup

## Граница

Одна доменная модель: корзина, заказ на выдачу, слот выдачи и учёт вместимости в `Expressa v1`.

## Сущности и value objects

### Entity `Cart`

| Поле | Тип семантики | Описание |
|---|---|---|
| `cartId` | Идентификатор | Идентификатор корзины customer до создания заказа. |
| `customerId` | Связь | Владелец корзины. |
| `items` | Состав | Набор выбранных строк корзины. |

### Entity `CartItem`

| Поле | Тип семантики | Описание |
|---|---|---|
| `menuItemId` | Связь | Выбранный товар. |
| `quantity` | Значение количества | Количество товара. |
| `selectedSize` | Значение размера | Размер напитка, если товар является напитком. |
| `selectedOptions` | Состав | Выбранные дополнительные опции. |

### Entity `PickupSlot`

| Поле | Тип семантики | Описание |
|---|---|---|
| `slotStart` | Временная точка | Начало слота выдачи. |
| `slotEnd` | Временная точка | Окончание слота выдачи. |
| `date` | Календарная дата | Дата слота; в v1 допустим только текущий день. |
| `capacityLimit` | Ограничение | Максимальное число активных заказов для слота. |
| `activeOrderCount` | Производное значение | Количество заказов, занимающих вместимость слота. |

### Entity `Order`

| Поле | Тип семантики | Описание |
|---|---|---|
| `orderId` | Идентификатор | Идентификатор заказа. |
| `customerId` | Связь | Customer, создавший заказ. |
| `items` | Состав | Строки заказа. |
| `pickupSlot` | Связь/встроенное значение | Выбранный слот выдачи. |
| `status` | Состояние | Статус жизненного цикла заказа. |
| `rejectionReason` | Атрибут | Причина отклонения заказа, если он отклонён. |
| `confirmedByBaristaId` | Audit-связь | Barista, подтвердивший заказ. |
| `readyMarkedByBaristaId` | Audit-связь | Barista, переведший заказ в `Готов к выдаче`. |
| `rejectedByBaristaId` | Audit-связь | Barista, отклонивший заказ. |

### Entity `OrderItem`

| Поле | Тип семантики | Описание |
|---|---|---|
| `menuItemId` | Связь | Товар, по которому создана строка заказа. |
| `quantity` | Значение количества | Количество в заказе. |
| `selectedSize` | Значение размера | Выбранный размер напитка, если применимо. |
| `selectedOptions` | Состав | Выбранные дополнительные опции. |
| `lineAmount` | Денежное значение | Сумма строки заказа. |

### Value object `OrderStatus`

Допустимые значения:

- `Создан`
- `Подтвержден`
- `Отклонен`
- `Готов к выдаче`
- `Закрыт`

### Value object `WorkingHours`

| Поле | Описание |
|---|---|
| `startTime` | Начало операционного окна. |
| `endTime` | Конец операционного окна. |

Значение по умолчанию: `09:00–20:00`.

## Отношения

- `Cart` принадлежит одному `customer`.
- `Order` создаётся из `Cart`.
- `Order` привязан к одному `PickupSlot`.
- `PickupSlot` определяется в пределах `WorkingHours`.
- `Order` может быть связан с `barista` через audit-атрибуты подтверждения, готовности и отклонения.

## Инварианты и ограничения

- Customer может просматривать и редактировать корзину до оформления заказа.
- Заказ создаётся только с выбором доступного слота выдачи.
- Новый заказ получает статус `Создан`.
- Допустимые статусы заказа ограничены перечислением `OrderStatus`.
- Только `barista` может отклонить заказ.
- При отклонении заказа причина отклонения обязательна.
- Заказы в статусах `Создан`, `Подтвержден`, `Готов к выдаче` занимают вместимость выбранного слота.
- Заказы в статусах `Отклонен`, `Закрыт` не занимают вместимость выбранного слота.
- Слоты формируются как интервалы по 10 минут внутри рабочих часов.
- Для выбора доступны только слоты текущего дня.
- Значение вместимости слота по умолчанию равно `5`.
- Оплата происходит офлайн при получении и не изменяет доменную модель v1 внутренней сущностью оплаты.

## Наблюдаемые пробелы

- Во входных материалах не определено, может ли customer изменять или отменять уже созданный заказ.
- Во входных материалах не определено, должен ли заказ хранить полные snapshot-данные названия товара и опций на момент оформления, или допустима ссылочная модель только на каталог.
- Не определено, должен ли выбор слота и создание заказа быть атомарным относительно проверки вместимости при конкурирующих действиях нескольких customer; это критично для контракта создания заказа и требует отдельного уточнения на архитектурном/системном уровне.
