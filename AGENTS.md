# AGENTS

## Старт работы

Перед началом любой работы в этом репозитории агент должен сначала прочитать [process/README.md](./process/README.md), затем [process/workflow.md](./process/workflow.md).

## Зачем это нужно

- `process/README.md` является входной точкой process-layer.
- `process/workflow.md` определяет переносимый рабочий маршрут.
- Этот файл фиксирует только project-specific правила и локальные override.

## Обязательные правила

### Чтение файлов

Агент начинает анализ, изменение файлов и постановку задач после чтения `process/README.md`, `process/workflow.md` и корневого `README.md`.

### Project-specific override

- Локальный project entrypoint для навигации по проектным артефактам: `README.md`.
- Карта терминов для русскоязычной проектной документации: `terms-map.md`.
- Агент должен использовать текущую карточку задачи, её `Ссылки на документы`, `Минимальный read set` и явно назначенные инструкции как маршрут чтения по умолчанию.
- Агент должен считать предыдущие задачи и `tasks/archive/` недопустимым источником формата, решений или эталона, если текущий маршрут чтения не ссылается на них явно.

### Правила нормализации инструкций

1. **Поведенческие требования (только позитивная форма)**
   Все функциональные и поведенческие требования формулируются только в позитивной форме:
   `Система должна …`

2. **Разделение ответственности**
   Ограничения не смешиваются с поведением и выносятся в отдельные разделы:
   - `Scope Constraints` — ограничения области выполнения
   - `Safety Constraints` — инварианты безопасности (инвариант — условие, которое не должно нарушаться ни при каких допустимых состояниях)

3. **Атомарность ограничений**
   Ограничения формулируются один раз, атомарно (одно правило — одно утверждение), без композиции и скрытых зависимостей.

4. **Запрет псевдопозитивных переформулировок**
   Запрещено переписывать запреты в псевдопозитивные или синонимические конструкции без прироста семантики.

5. **Отсутствие дублирования**
   Запрещено дублировать одно и то же требование в разных формулировках (позитивной, негативной или синонимической).

6. **Устранение редундантности**
   Удаление допустимо только при доказанной редундантности (редундантность — отсутствие влияния на наблюдаемое поведение системы при любом допустимом сценарии).

7. **Требование семантической значимости**
   Каждая инструкция должна добавлять новую проверяемую семантику; иначе она подлежит удалению.
