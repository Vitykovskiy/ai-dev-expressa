# Карта проверок и поставки

## Назначение

Эта карта фиксирует фактическое устройство сквозных проверок, инфраструктурных артефактов, конвейера и маршрутов развёртывания. Её читают задачи `QA-*`, `DO-*` и задачи, которые меняют путь тестирования, сборки, дымовой проверки или развёртывания.

## Контуры и ответственность

| Контур | Путь | Ответственность | Основные зависимости |
| --- | --- | --- | --- |
| Сквозные проверки | `e2e/` | Playwright-наборы `FEATURE-*`, которые поддерживает `QA` и запускает против уже развернутого `test`-окружения VPS | `@playwright/test`, `apps/server`, `apps/backoffice-web` |
| Infra | `infra/` | Generic контейнерный маршрут, шаблоны окружения, VPS-скрипт развёртывания и Node-скрипт дымовой проверки | `Docker`, `Docker Compose`, исполняемые контуры |
| Конвейер | `.github/workflows/` | Проверка запроса на слияние non-e2e набором, автоматическое развёртывание `main` в `test` и ручной выпуск в `production` | `GitHub Actions`, `infra/` |

## Текущий состав

- `e2e/feature-001` — Playwright-набор `QA-001`, который проверяет доступ `administrator` и поддерживается `QA` для запуска против уже развернутого `test`-окружения VPS.
- `e2e/feature-002` — Playwright-набор `QA-002`, который проверяет сквозное управление каталогом меню `administrator` на уже развернутом `test`-окружении VPS.
- `infra/docker/compose.yml` — generic контейнерный маршрут.
- `infra/docker/.env.example` и `infra/docker/.env.server.example` — шаблоны окружения контейнерного маршрута.
- `infra/scripts/deploy.sh` — воспроизводимое развёртывание на VPS.
- `infra/scripts/smoke-backoffice-access.mjs` — feature-специфичная дымовая проверка доступа во внутренний административный контур, запускаемая из generic-маршрута.
- `.github/workflows/ci.yml` — модульные проверки, сборка, валидация контейнерного маршрута и локальная дымовая проверка.
- `.github/workflows/deploy.yml` — автоматическое развёртывание `test` по `push` в `main`, ручной выпуск в `production` и post-deploy дымовая проверка.
- GitHub branch protection для `main` — обязательные статусы `quality` и `deployment-validation`, которые формирует только `ci.yml`.

## Фактический путь проверок и поставки

1. Локально разработчик поднимает нужные исполняемые контуры, выполняет модульные тесты и локальные проверки по задаче.
2. `ci.yml` на запросе на слияние выполняет только non-e2e набор: `npm ci`, модульные тесты `apps/server` и `apps/backoffice-web`, сборку обеих частей, валидацию контейнерного маршрута и локальные дымовые проверки рабочего режима и test-режима.
3. `deploy.yml` на каждом merge в `main` повторяет тот же non-e2e набор, затем разворачивает постоянное `test`-окружение VPS через `infra/scripts/deploy.sh` и `infra/docker/compose.yml` с `DISABLE_TG_AUTH=true`.
4. После завершения развёртывания `QA` запускает e2e на уже развернутом `test`-окружении VPS командой `E2E_BASE_URL=<BACKOFFICE_PUBLIC_URL> npm run test:e2e`, фиксирует результат прогона и оформляет воспроизводимые дефекты.
5. `deploy.yml` для `production` запускается только вручную и не включает e2e; после выпуска подтверждается post-deploy дымовая проверка рабочего режима.

## Feature-срезы

### `FEATURE-001`: доступ во внутренний административный контур

- Дымовая проверка остаётся feature-специфичной и подтверждает критический сценарий доступа во внутренний административный контур.
- `ci.yml` и `deploy.yml` не запускают Playwright e2e для `FEATURE-001`; они ограничиваются локальной или post-deploy дымовой проверкой.
- Generic VPS-маршрут синхронизирует репозиторий на сервер, рендерит env-файлы, запускает `infra/scripts/deploy.sh` и завершает выпуск дымовой проверкой; для `test` фиксируется `DISABLE_TG_AUTH=true` и прямой доступ по URL без Telegram.
- После автоматического развёртывания `test` `QA` выполняет e2e для проверки browser-only test-пути доступа и отказа явного рабочего bootstrap без Telegram-контекста, затем фиксирует дефекты вне маршрута непрерывной интеграции и доставки.

### `FEATURE-002`: управление каталогом меню

- Использует существующие команды сборки, модульных тестов и общий маршрут развёртывания.
- E2e-набор поддерживается `QA` и выполняется против уже развернутого `test`-окружения VPS после завершения `deploy.yml`.
- Отдельные workflow, env-файлы и скрипты развёртывания для текущей архитектурной рамки не требуются.
- Отдельная задача `DO-*` не выделяется, пока фича укладывается в существующие исполняемые контуры, переменные окружения и generic маршрут развёртывания.

## Команды запуска и проверки

- Установка зависимостей: `npm install`.
- E2e-проверка на VPS: `E2E_BASE_URL=<BACKOFFICE_PUBLIC_URL> npm run test:e2e`.
- Локальная e2e-проверка рабочего пути: `npm run test:e2e`.
- Локальная e2e-проверка test-пути: `E2E_ACCESS_MODE=test npm run test:e2e`.
- Проверка контейнерного маршрута: `npm run deploy:config`.
- Дымовая проверка `FEATURE-001`: `npm run smoke:backoffice-access`.
- Сборка серверной части: `npm run build --workspace @expressa/server`.
- Сборка клиентской части: `npm run build --workspace @expressa/backoffice-web`.
- Модульные и HTTP-тесты серверной части: `npm run test --workspace @expressa/server`.
- Модульные тесты клиентской части: `npm run test --workspace @expressa/backoffice-web`.
- Маршруты развёртывания для конкретных окружений читаются из `deployment-map.md`.

## Когда обновлять карту

- Появился новый e2e-набор или изменился путь его запуска.
- Изменился compose-файл, шаблон окружения, скрипт развёртывания или дымовая проверка.
- Изменились workflow, этапы конвейера, условия запуска или post-deploy проверка.
- Изменился маршрут развёртывания, отката или порядок восстановления.
