# Правила дерева задач для `Sprint -> Feature`

## Назначение

Эта инструкция определяет, как системный аналитик должен формировать backlog-модель для одной единицы поставки при подготовке задач для разработки.

## Правила

- Одна родительская задача `Sprint-*` должна представлять ровно одну delivery unit и ничего за ее пределами.
- `Sprint-*` является координационной границей backlog, а не прямой задачей на один change set.
- Delivery unit для `Sprint-*` — это одна завершенная, работоспособная, тестируемая и демонстрируемая поставка для customer или stakeholder.
- Системный аналитик создает или обновляет только `Sprint-*`-карточку.
- Системный аналитик не должен заранее создавать `AR/FE/BE/DO/QA` child-задачи и не должен декомпозировать фичи на role-specific работу.
- Архитектор после фиксации стека, кодстайла и архитектурной карты приложения создает backlog фич `FEATURE-*` внутри `Sprint-*`.
- `FEATURE-*` должна быть минимальным independently reviewable vertical slice внутри спринта.
- Если фичу нельзя развернуть, проверить отдельным smoke/scenario и показать как самостоятельный результат, она декомпозирована недостаточно.
- Архитектор не должен заранее создавать `AR/FE/BE/DO` для всех фич спринта; эти задачи создаются только для той `FEATURE-*`, которая реально стартует в разработку.
- При старте конкретной фичи архитектор создает отдельную `AR-*`-задачу этой фичи и только затем `FE/BE/DO-*` при необходимости.
- `QA-*` по умолчанию создается на уровне `Sprint-*` и фиксирует финальную приемку собранного инкремента из завершенных фич.
- Пока хотя бы одна обязательная `FEATURE-*` в спринте не завершена, `Sprint-*` остается в статусе `В работе`.
- `FEATURE-*` переходит в статус `Ожидает тестирования` только после завершения своих `AR/FE/BE/DO` и готовности feature smoke.
- `Sprint-*` переходит в статус `Ожидает тестирования` только после завершения обязательных фич и готовности финальной sprint QA.
- Нельзя объединять несколько delivery unit в одну карточку `Sprint-*`, даже если они относятся к одному бизнес-сценарию или были выделены в рамках одной аналитической сессии.

## Примечания

- Аналитическая декомпозиция системы на меню, слоты, уведомления, доступ и другие системные срезы не равна backlog-декомпозиции на `Sprint-*` и `FEATURE-*`.
- `Sprint-*` является координационной границей для одной единицы поставки, а `FEATURE-*` — минимальной границей отдельно проверяемой реализации внутри этой поставки.
- Канонический пример первой фичи: рабочее окружение поднято, backend отвечает минимальным endpoint, frontend ходит в backend, devops обеспечивает локальный/test runtime, а smoke подтверждает связку `client -> server`.
