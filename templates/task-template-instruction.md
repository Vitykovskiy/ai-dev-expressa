# Инструкция к шаблону задачи

## Назначение

Этот шаблон используется для постановки задач по проекту. Карточка должна однозначно определять, кто выполняет задачу, что является результатом и на какие документы нужно опираться.

## Правила заполнения

- `Идентификатор` задается в формате `<тип или сокращение роли>-<инкремент>`.
- Инкремент ведется отдельно по каждому типу карточки или роли и заполняется с лидирующими нулями: `001`, `002`, `003`.
- `SPRINT-*` — координационная карточка спринта. `FEATURE-*` — карточка одной завершенной фичи и единственная карточка единицы поставки.
- `Родительская задача` обязательна для дочерних задач и содержит идентификатор родительской `SPRINT-*` или `FEATURE-*` карточки. Для верхнеуровневой карточки спринта и аналитических задач используется значение `нет`.
- `Заголовок` формулируется кратко и в терминах ожидаемого результата.
- `Описание` должно содержать суть работы, ограничения, важные условия и ожидаемый итог.
- `Единица поставки` обязательна для `FEATURE-*` и для всех контурных задач под фичей. Значением должен быть идентификатор фичи, например `FEATURE-001`.
- Для `SPRINT-*`, аналитических задач и задач архитектора по декомпозиции спринта на фичи допустимо значение `n/a`.
- Для задач в потоке поставки единица поставки должна означать завершенный, работоспособный, тестируемый и демонстрируемый результат, а не внутренний технический срез.
- `Роль` выбирается только из утвержденного списка ролей.
- `Изменяемый контур` фиксирует, в каком контуре находится результат задачи: `delivery-unit`, `architecture`, `frontend`, `backend`, `devops`, `qa` или `n/a`.
- `SPRINT-*` по умолчанию используют `n/a`, `FEATURE-*` — `delivery-unit`, а ролевые задачи используют свой профильный контур.
- `Приоритет` выбирается только по утвержденной шкале `Низкий`, `Средний`, `Высокий`, `Критический`.
- В блоке `Ссылки на документы` ссылки группируются по категориям `Системные артефакты`, `Архитектурные артефакты`, `Бизнес-артефакты` и `Дополнительные материалы`.
- `Системные артефакты` содержат только релевантные файлы из `docs/system/`, если они нужны исполнителю.
- `Архитектурные артефакты` содержат только релевантные файлы из `docs/architecture/`.
- `Контурная карта` обязательна для `FE-*`, `BE-*`, `DO-*`, `QA-*` и для feature-задач, которые заранее затрагивают известные контуры.
- Для задачи в одном контуре указывается одна карта из `docs/architecture/application-map/`; для задачи на несколько контуров перечисляются все нужные контурные карты.
- Полный `docs/architecture/application-map.md` указывается только для архитектурных задач, которые меняют индекс карт, список контуров или маршрут навигации.
- `Бизнес-артефакты` используются только там, где исполнитель действительно опирается на исходный бизнес-контекст; если они не нужны, указывается `не требуются`.
- `Дополнительные материалы` используются только для точечных ссылок вне системных и архитектурных артефактов; если они не нужны, указывается `не требуются`.
- `Минимальный read set` является обязательным рабочим маршрутом чтения, а не справочной заметкой.
- `Минимальный read set` перечисляет только тот минимальный набор документов и контурных карт, который действительно нужен исполнителю.
- Для обычных `FE-*` и `BE-*` задач `Минимальный read set` не должен ссылаться на полный `docs/architecture/application-map.md`; вместо него используется нужная контурная карта.
- `Ожидаемый результат для ревью` фиксирует проверяемый результат, а не перечень действий.
- `Проверки` фиксируют конкретные проверки: модульные тесты, e2e-тесты, дымовая проверка, сценарий приёмки, pipeline-validation.
- `Обновление карты приложения` обязательно указывает, должен ли исполнитель обновить `docs/architecture/application-map.md` и связанные навигационные файлы.
- Если задача меняет факты конкретного контура, в `Обновление карты приложения` указывается соответствующая контурная карта; индекс `docs/architecture/application-map.md` обновляется только при изменении навигации или списка контуров.

## Градация приоритетов

| Приоритет     | Когда использовать                                                            | Что означает для очереди                         |
| ------------- | ----------------------------------------------------------------------------- | ------------------------------------------------ |
| `Критический` | Запуск продукта или продолжение потока поставки невозможны без этой задачи     | Брать в работу раньше всех остальных задач со статусом `Готова к работе` |
| `Высокий`     | Задача нужна для основного пути v1 или разблокирует несколько следующих задач | Брать в работу в первой волне после критических  |
| `Средний`     | Задача завершает подтверждённый scope, но не мешает старту ядра продукта      | Брать после задач, открывающих основной поток    |
| `Низкий`      | Локальное улучшение или изолированный результат без блокировки других задач   | Брать в работу в последнюю очередь               |

- `Статус` выбирается только из утвержденного списка статусов.
- В блоке `Ссылки на документы` указываются только релевантные документы, необходимые для выполнения задачи.
- Для `SA-*` и `SPRINT-*` допустимо использовать `Бизнес-артефакты` как исходный материал анализа и подготовки спринта.
- Для `FEATURE-*`, `AR-*`, `FE-*`, `BE-*`, `DO-*`, `QA-*` основой карточки должны быть релевантные `Системные артефакты` и `Архитектурные артефакты`.
- Для `FEATURE-*`, `AR-*`, `FE-*`, `BE-*`, `DO-*`, `QA-*` значение `Бизнес-артефакты` по умолчанию — `не требуются`.
- Исключение для бизнес-артефакта в `FEATURE-*`, `AR-*`, `FE-*`, `BE-*`, `DO-*`, `QA-*` допустимо только если нужный факт ещё не перенесён в `docs/system/`; в таком случае ссылка должна быть единичной и сопровождаться кратким пояснением, почему без неё задача неисполнима.
- Для `FEATURE-*`, `AR-*`, `FE-*`, `BE-*`, `DO-*`, `QA-*` блок `Минимальный read set` не должен содержать ссылки на `docs/business/`, кроме такого явно отмеченного исключения.
- В `Критериях готовности` фиксируется проверяемый результат, а не процесс выполнения.

## Типы карточек и сокращения ролей для идентификатора

- `SPRINT` — координационная карточка спринта
- `FEATURE` — карточка единицы поставки
- `AR` — Архитектор
- `BA` — Бизнес аналитик
- `SA` — Системный аналитик
- `DO` — Девопс
- `FE` — Фронтенд
- `BE` — Бэкенд
- `QA` — Тестирование

## Допустимые роли

- Архитектор
- Разработка
- Бизнес аналитик
- Системный аналитик
- Девопс
- Фронтенд
- Бэкенд
- Тестирование

## Допустимые статусы

- Готова к работе
- В работе
- Ожидает тестирования
- В тестировании
- Выполнена

## Примеры идентификаторов

- `SPRINT-001` — первый спринт в backlog
- `FEATURE-001` — первая фича как единица поставки
- `AR-001` — первая задача для архитектора
- `BA-001` — первая задача для бизнес-аналитика
- `SA-003` — третья задача для системного аналитика
- `FE-012` — двенадцатая задача для фронтенда
