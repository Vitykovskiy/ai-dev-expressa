# Карта терминов и русских аналогов

Этот файл фиксирует англицизмы и смешанные формулировки, которые уже встречаются в документации проекта, и задает для них рекомендуемые русские аналоги.

Названия технологий, продуктов, протоколов, переменных окружения и путей в репозитории не переводятся: `Vue`, `Docker`, `Telegram Bot API`, `DATABASE_URL`, `apps/api` и т.д.

| Требуется заменить | Термин в документации | Рекомендуемый русский аналог | Комментарий |
| --- | --- | --- | --- |
| Нет | `frontend` | клиентская часть | Использовать в описании пользовательских интерфейсов и web-клиентов. |
| Нет | `backend` | серверная часть | Использовать в описании API, доменной логики и интеграций. |
| Нет | `backoffice` | внутренний административный контур | Подходит для общего описания служебного интерфейса вне имен собственных. |
| Да | `customer bot` | клиентский бот | Для Telegram-бота, через которого работает клиент. |
| Да | `backoffice bot` | служебный бот | Для Telegram-бота внутренних ролей. |
| Да | `WebApp` | веб-приложение | Предпочтительно в обычном русском тексте. |
| Да | `customer-facing` | клиентский | Если речь идет об интерфейсе или сценарии для клиента. |
| Да | `entrypoint` | точка входа | Для описания запуска модуля, сервиса или приложения. |
| Да | `runtime contour` | исполняемый контур | Для описания отдельно запускаемой части системы. |
| Нет | `framework` | каркас разработки | Использовать в общих архитектурных формулировках. |
| Да | `persistence layer` | слой хранения данных | Для слоя доступа к БД и постоянному хранению. |
| Да | `deployment tool` | инструмент развёртывания | Для средств публикации и выката. |
| Да | `deployment path` | маршрут развёртывания | Когда описывается путь изменения до целевого окружения. |
| Да | `deploy` | развёртывание | Глагольную форму лучше писать по-русски: "выполнить развёртывание". |
| Нет | `smoke-check` | дымовая проверка | Базовая проверка работоспособности после сборки или развёртывания. |
| Да | `rollback` | откат | Возврат к предыдущему рабочему состоянию. |
| Да | `restore path` | порядок восстановления | Если нужен путь возврата после сбоя. |
| Да | `rollback strategy` | стратегия отката | Формулировка для документации по рискам и релизам. |
| Нет | `pipeline` | конвейер | Для CI/CD и других автоматизированных цепочек действий. |
| Да | `pipeline stages` | этапы конвейера | Для детализации шагов процесса. |
| Нет | `CI/CD` | непрерывная интеграция и доставка | Допустимо оставлять аббревиатуру, но при первом упоминании лучше раскрывать смысл. |
| Да | `pull request` | запрос на слияние | Для процессов ревью и интеграции изменений. |
| Да | `merge readiness` | готовность к слиянию | Статус готовности ветки к объединению. |
| Да | `review gates` | критерии допуска к ревью | Проверки, без которых задача не должна идти на ревью. |
| Да | `quality gates` | критерии качества | Обязательные условия приемки качества. |
| Да | `outcome` | законченный результат | Для описания проверяемого результата задачи. |
| Да | `change set` | набор изменений | Для группы правок, рассматриваемых вместе. |
| Да | `child task` | дочерняя задача | Для задач, связанных с родительской delivery-единицей. |
| Да | `unit tests` | модульные тесты | Предпочтительный вариант в текстовой документации. |
| Да | `shared types` | общие типы | Для типов, используемых несколькими контурами. |
| Да | `shared package` | общий пакет | Для переиспользуемого пакета монорепозитория. |
| Да | `shared contracts` | общие контракты | Для общих схем обмена и интерфейсов. |
| Да | `use case` | вариант использования | Для системного поведения и сценариев взаимодействия. |
| Да | `orchestration` | оркестрация | Если речь о согласовании нескольких шагов или модулей. |
| Да | `env vars` | переменные окружения | Для параметров запуска и конфигурации. |
| Да | `env/config` | окружение и конфигурация | Когда важны оба аспекта одновременно. |
| Да | `dependency edge` | ребро зависимости | Для описания ребра зависимости между модулями. |
| Да | `transport/integration layer` | слой транспорта и интеграций | Для внешних интерфейсов и способов обмена данными. |
| Да | `adapter layer` | слой адаптеров | Для прослойки между доменной логикой и внешними интерфейсами. |
| Нет | `composables` | композиционные функции | Предпочтительный русский вариант для Vue-описаний. |
| Нет | `stores` | хранилища состояния | Для Pinia и аналогичных механизмов хранения состояния. |
| Да | `build` | сборка | Для результата и процесса построения артефактов. |
| Да | `install dependencies` | установка зависимостей | Для шагов CI/CD и локального запуска. |
| Да | `runtime artifacts` | исполняемые артефакты | Для сборок, которые идут в публикацию или развёртывание. |
| Да | `publish container images` | публиковать контейнерные образы | Для шагов конвейера поставки. |
| Да | `secret` | секрет | Для скрытых ключей, токенов и чувствительных параметров. |
| Да | `coding standard` | стандарт кодирования | Для правил оформления и написания кода. |
| Да | `deployment guidance` | инструкция по развёртыванию | Для навигации по выкату и окружениям. |
