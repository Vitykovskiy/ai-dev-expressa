# Инструкция к шаблону задачи

## Назначение

Этот шаблон используется для постановки задач по проекту. Карточка должна однозначно определять, кто выполняет задачу, что является результатом и на какие документы нужно опираться.

## Правила заполнения

- `Идентификатор` задается в формате `<сокращение роли>-<инкремент>` для role-specific задач и в специальных форматах `Sprint-<инкремент>` / `FEATURE-<инкремент>` для backlog-карточек.
- Инкремент ведется отдельно по каждой роли и заполняется с лидирующими нулями: `001`, `002`, `003`.
- `Родительская задача` обязательна для child-задач backlog-модели и содержит идентификатор родительской карточки: `FEATURE-*` ссылается на `Sprint-*`, `AR/FE/BE/DO-*` ссылаются на `FEATURE-*`, а финальная `QA-*` по умолчанию ссылается на `Sprint-*`. Для родительской sprint-карточки и аналитических задач используется значение `нет`.
- `Заголовок` формулируется кратко и в терминах ожидаемого результата.
- `Описание` должно содержать суть работы, ограничения, важные условия и ожидаемый итог.
- `Единица поставки` обязательна для задач, которые передаются в разработку, тестирование или иной delivery-поток. `Sprint-*` ссылается на одну delivery unit `DU-*`, а `FEATURE-*` и role-specific child-задачи — на под-единицу вида `DU-01.F01`. Для аналитических задач, которые не выпускают delivery unit напрямую, допустимо значение `n/a`.
- Для задач в delivery-потоке единица поставки должна означать завершенный, работоспособный, тестируемый и демонстрируемый результат. Для `FEATURE-*` это отдельный vertical slice внутри спринта, а не абстрактный технический срез без собственного smoke/scenario.
- `Роль` выбирается только из утвержденного списка ролей.
- `Изменяемый контур` фиксирует, в каком контуре находится результат задачи: `sprint`, `feature`, `architecture`, `frontend`, `backend`, `devops`, `qa` или `n/a`.
- `Приоритет` выбирается только по утвержденной шкале `Низкий`, `Средний`, `Высокий`, `Критический`.
- `Минимальный read set` перечисляет только тот набор документов, который действительно нужен исполнителю.
- `Ожидаемый результат для ревью` фиксирует reviewable outcome, а не перечень действий.
- `Проверки` фиксируют конкретные проверки: unit tests, smoke-check, сценарий приёмки, pipeline-validation.
- `Обновление карты приложения` обязательно указывает, должен ли исполнитель обновить `docs/architecture/application-map.md` и связанные навигационные файлы.
- Если результат нельзя развернуть, проверить отдельным smoke/scenario или отревьюить как самостоятельный vertical slice, задача декомпозирована недостаточно.

## Градация приоритетов

| Приоритет     | Когда использовать                                                            | Что означает для очереди                         |
| ------------- | ----------------------------------------------------------------------------- | ------------------------------------------------ |
| `Критический` | Запуск продукта или продолжение delivery-потока невозможны без этой задачи    | Брать в работу раньше всех остальных ready-задач |
| `Высокий`     | Задача нужна для основного пути v1 или разблокирует несколько следующих задач | Брать в работу в первой волне после критических  |
| `Средний`     | Задача завершает подтверждённый scope, но не мешает старту ядра продукта      | Брать после задач, открывающих основной поток    |
| `Низкий`      | Локальное улучшение или изолированный результат без блокировки других задач   | Брать в работу в последнюю очередь               |

- `Статус` выбирается только из утвержденного списка статусов.
- В блоке `Ссылки на документы` указываются только релевантные документы, необходимые для выполнения задачи.
- В `Критериях готовности` фиксируется проверяемый результат, а не процесс выполнения.

## Сокращения ролей для идентификатора

- `AR` — Архитектор
- `DEV` — Разработка
- `BA` — Бизнес аналитик
- `SA` — Системный аналитик
- `DO` — Девопс
- `FE` — Фронтенд
- `BE` — Бэкенд
- `QA` — Тестирование

Специальные backlog-префиксы:

- `Sprint` — родительская карточка спринта, соответствующая одной delivery unit
- `FEATURE` — фича внутри конкретного спринта

## Примеры идентификаторов

- `AR-001` — первая задача для архитектора
- `Sprint-001` — первый спринт delivery-backlog
- `FEATURE-001` — первая фича внутри общего backlog
- `BE-001` — первая задача для бэкенда
- `BA-001` — первая задача для бизнес-аналитика
- `SA-003` — третья задача для системного аналитика
- `FE-012` — двенадцатая задача для фронтенда

## Модель backlog-потока

- `Sprint-*` — родительская карточка одного спринта, соответствующего одной delivery unit.
- `FEATURE-*` — backlog фич внутри одного `Sprint-*`.
- Архитектор сначала создает backlog фич внутри `Sprint-*`, а не разрезает весь спринт сразу на `AR/FE/BE/DO`.
- При старте конкретной `FEATURE-*` архитектор создает отдельную `AR-*` карточку этой фичи и только затем `FE/BE/DO-*` по необходимости.
- `QA-*` по умолчанию создается на уровне `Sprint-*` для финальной приемки собранного инкремента.
- Одна child-задача должна давать один reviewable outcome в одном контуре.
- Канонический пример первой фичи: backend поднимает базовый runtime и минимальный endpoint, frontend поднимает базовый клиент и умеет сходить в backend, devops поднимает test/local runtime и smoke-маршрут, результатом является рабочая связка `client -> server`.
