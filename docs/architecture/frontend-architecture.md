# Стандарт архитектуры клиентской части на `Vue 3` и `TypeScript`

## Назначение

Этот документ задаёт обязательный стандарт для клиентской части на `Vue 3` и `TypeScript`. Он определяет канонический каркас каталогов, границы слоёв, правила размещения логики, требования к общим типам и модульным тестам, а также критерии допуска к ревью.

Фактическая карта приложения, реальные точки входа, названия модулей и текущие зависимости фиксируются в `application-map.md`. Этот документ описывает не текущее состояние кода, а правила, которым это состояние должно соответствовать.

## Область действия

- Стандарт обязателен для всех Vue-приложений монорепозитория.
- Если для задачи требуется отклонение от стандарта, оно должно быть зафиксировано отдельным архитектурным решением до начала реализации.
- Любое изменение структуры клиентской части должно проверяться на соответствие этому документу и `application-map.md`.

## Базовые принципы

- Клиентская часть строится на жёстком разделении слоёв и прямом потоке данных сверху вниз.
- Один модуль не должен одновременно владеть маршрутизацией, состоянием и транспортом.
- Общая логика выносится только при наличии повторяемой роли и понятной границы ответственности.
- Локальные упрощения не должны создавать отдельные "мини-архитектуры" внутри одной фичи.
- Прямой `fetch` и другие API-вызовы вне слоя транспорта запрещены.

## Канонический каркас `src/`

```text
src/
  App.vue
  main.ts
  types.ts
  components/
  composables/
  layouts/
  pages/
  router/
  services/
  stores/
  styles/
```

### Роли каталогов

| Каталог или файл | Роль | Запрещено |
| --- | --- | --- |
| `App.vue` | Корневой каркас клиентской части и подключение верхнего layout | HTTP, предметные вычисления, политика доступа |
| `main.ts` | Точка входа, сборка приложения, подключение плагинов и router | Предметные правила и ветвление доступа |
| `types.ts` | Локальные типы конкретного приложения | Дублирование общих контрактов |
| `pages/` | Сборка экранов из подготовленных данных и переиспользуемых частей | Прямые API-вызовы и тяжёлая оркестрация |
| `layouts/` | Каркас разделов, shell и общие области встраивания | Транспорт, политика доступа, источник истины по данным |
| `router/` | Маршруты, route guards, метаданные маршрутов и навигационная конфигурация | HTTP, хранение пользовательского состояния, DOM-логика |
| `stores/` | Долгоживущее состояние приложения и сценарии его обновления | Прямая работа с DOM и шаблонами |
| `composables/` | Переиспользуемая UI-логика, локальная оркестрация и склейка источников данных | Подмена собой `store`, `page` и `service` одновременно |
| `services/` | Транспорт, адаптеры окружения, хранение и нормализация внешних данных | UI-состояние и политика маршрутизации |
| `components/` | Переиспользуемые визуальные блоки | Знание о конкретном API, маршруте или доступе |
| `styles/` | Глобальные стили, токены и темы | Логика приложения |

## Границы ответственности

### `pages/`

- Страница собирает экран из layout, компонентов, composable-функций и stores.
- Страница может выбирать, какие данные показывать, но не должна реализовывать транспорт и предметные правила.
- Если в странице появляются API-вызовы, разветвлённая политика доступа или сложная оркестрация, код находится не в том слое.

### `layouts/`

- Layout задаёт каркас раздела, общие зоны и slot-структуру.
- Layout может принимать уже рассчитанные флаги и данные.
- Layout не должен становиться источником истины по доступу, маршрутам или бизнес-данным.

### `router/`

- В `router/` живут маршруты, route guards, навигационные метаданные и сборка навигации.
- Guard использует уже существующие stores, composable-функции и типы, но не содержит транспорт.
- Логика допуска к маршруту допустима только в `router/`, а не в шаблонах и не в сервисах.

### `stores/`

- Store отвечает за долгоживущее состояние и сценарии его изменения.
- Store может обращаться к сервисам, если это часть сценария обновления состояния.
- Store не должен знать о DOM, шаблоне компонента или внутреннем устройстве конкретной страницы.

### `composables/`

- Composable-функция нужна для переиспользуемой UI-логики и локальной оркестрации.
- Сюда выносится подготовка представления, реактивные зависимости и склейка нескольких источников данных.
- Если composable начинает знать про router, store, DOM и API сразу, его нужно декомпозировать.

### `services/`

- Сервисы описывают работу с внешними интерфейсами: HTTP, `localStorage`, Telegram WebApp, окружение и нормализацию ответов.
- Сервисы возвращают данные и ошибки в форме, пригодной для вышележащих слоёв.
- Сервисы не хранят UI-состояние и не решают, что рендерить или куда перенаправлять пользователя.

### `components/`

- Компонент переиспользуется как визуальный строительный блок.
- Компонент получает данные и обработчики через props, emits или slot.
- Компонент не должен инициировать маршрутную политику, вычислять права доступа или обращаться к транспорту.

## Правила зависимостей

- `pages/` и `layouts/` могут использовать `components/`, `composables/`, `stores/`, типы и маршрутные константы.
- `router/` может использовать типы, `stores/`, `composables/` и чистые функции подготовки навигации.
- `stores/` могут использовать `services/` и типы.
- `composables/` могут использовать `stores/`, `services/` и типы, если это не размывает их роль.
- `services/` могут использовать только типы, внешние библиотеки и другие сервисы с более низким уровнем абстракции.
- Нельзя складывать общий код в случайные `utils`, `helpers`, `misc` или аналогичные каталоги без явной архитектурной роли.

## Правила типов

- Общие межконтурные типы, DTO и контракты лежат только в `packages/shared-types/`.
- Типы конкретного Vue-приложения лежат в `src/types.ts` или рядом со слоем, которому они принадлежат.
- Не дублировать API-контракты вручную между приложениями.
- Ручное дублирование серверных контрактов внутри клиентской части запрещено.
- Локальные типы должны описывать клиентское состояние, конфигурации представления и внутренние соглашения приложения, а не переопределять серверный контракт.

## Правила для одиночных файловых компонентов

- В одиночном файловом компоненте обязателен порядок `template -> script -> style`.
- Компонент должен оставаться тонким: подготовка данных, преобразования, ветвления доступа и побочные эффекты выносятся в `stores/`, `composables/` или `services/`.
- Шаблон должен читать уже подготовленные данные, а не вычислять политику приложения на лету.
- Если логика в `script` перестаёт быть понятной без длинной цепочки вычислений и `watch`, её нужно выносить в другой слой.

## Модульные тесты

- Обязательны модульные тесты для `stores/`, `composables/`, форматтеров, адаптеров API, валидаторов и любой UI-логики с ветвлением.
- Route guards должны быть покрыты проверками, если в них есть нетривиальная логика доступа.
- Исключение из модульных тестов допустимо только для тривиальной разметки или тривиальной конфигурации и должно быть явно указано в задаче или запросе на слияние.

## Критерии допуска к ревью

- Задача считается готовой к ревью только если обновлены релевантные документы из `docs/architecture/`.
- Ревьюер должен видеть один завершённый результат в одном контуре.
- Для каждого нового модуля выбран корректный слой.
- Зависимости идут в допустимом направлении и не нарушают границы слоёв.
- Логика доступа, маршрутизации, транспорта и состояния не смешана в одном модуле.
- Ветвящаяся логика вынесена из шаблонов.
- Модульные тесты покрывают нетривиальную логику `stores/`, `composables/`, `services/` и route guards.

## Связь с `application-map.md`

- Если изменение затрагивает структуру каталогов, точки входа, общие пакеты, окружение, конфигурацию или тестовый путь, нужно обновить `application-map.md`.
- Если изменение делает корневую навигацию по репозиторию неполной, нужно обновить `README.md`.
